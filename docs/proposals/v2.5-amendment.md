## In plain terms (read this first)

Three small fixes to how a feature goes from contract → built → shipped:

1. **Don't let the builder cheat its own test.** Write the tests *before* building and lock
   them, so the builder can't quietly weaken the ruler it's measured by. And "I ran out of
   time" never counts as "I'm done" — it hands back honestly as *not finished*.

2. **Repeated corrections become house rules.** The first time you fix something, it's a
   one-off. The third time you fix the *same* thing across different features, that's a rule —
   it should graduate from "a thing you keep saying" into a written house rule the reviewer
   enforces, so you never have to say it a fourth time. Only when it actually recurs, and only
   when you say so — one-off taste stays taste.

3. **The corrections log that makes #2 possible.** Today we throw away a feature's working
   folder when it ships, which deletes the record of what you corrected. So we save corrections
   into the permanent record on the way out, noting *who* caught each one (you, the reviewer,
   or the builder). That's the memory that lets you notice "I've now said this four times."

Deferred on purpose: the "you promised X but nothing checks it" warning (build it later as part
of a review system), and the whole cross-feature/cross-project learning machine (no data yet —
don't build for a corpus that doesn't exist).

## The changes

### C3 — Don't let the builder cheat its own test

**The hole.** The contract currently lets "done" rest on *the builder's own say-so* where no
command fits ("a runnable command is the strongest form, not the required form"). That's the
exact surface a builder games — faking a pass, hard-coding the answer, or "giving up into
done" when its budget drains.

**The fix (three parts):**

1. **Tests are authored before the build, and the builder doesn't author the ruler it's
   measured by.** Acceptance checks are written and committed when the contract is ratified
   (planner or decider), *before* the build run. If a feature's provable checks are missing,
   they're written at ratification or the fresh reviewer audits coverage *before* the inner
   loop runs — never generated by the builder mid-loop. *(This closes the "builder writes
   itself a lenient test" loophole both review passes flagged.)*

2. **Locked for the run.** Once ratified, the checks are frozen relative to the builder. A
   check edited during the build without re-ratifying is a **defect, not a pass.** *(Freeze
   line = the ratification / Spec-PR merge the canon already treats as the ratification event.
   Enforcement is a `specline diff` rule, marked **(planned)** — it needs commit history, same
   class as the existing `stale_after`-moved check.)*

3. **Running out of time is a failure, not an exit.** Budget exhaustion hands back
   *unconverged* and escalates to the human. The cheapest way out of the build loop must be a
   real pass, never a drained budget.

### D — Repeated corrections become house rules

Not a new layer — your house rules (`conventions/`, `technical/`) already exist, and the fresh
reviewer already grades against them. This just names the *lifecycle*:

> **Promotion.** A correction that **recurs across distinct features** may graduate one step:
> from taste-in-your-head → a written, cited house rule the reviewer enforces → eventually an
> automated check. **Gated by recurrence + your sign-off.** Never automatic. A one-off
> correction **stays a one-off** — promoting it early just buys rigidity. What makes a promoted
> rule different from a flat note in `CLAUDE.md`: it's *cited at acceptance* and *graded by the
> fresh reviewer* — enforced, not merely written down.

### The corrections log (the artifact that makes D work)

A **required, shape-checked** section of the loop's status file:

```
## Corrections
<what was corrected> — <altitude: provable|judgeable|tasteable> — <who caught it: implementer|reviewer|decider>
```

- **Who-caught-it matters at the promotion gate:** a correction *you* made is a policy signal;
  one the builder caught itself may be a single-feature observation. Different weight when
  deciding whether it becomes a rule.
- **It survives the feature.** On graduation, the corrections fold into the permanent knowledge
  doc *before* the spec folder is deleted — so cross-feature recurrence is observable instead
  of shredded. (For now you read recurrence across knowledge docs yourself; a tool can help
  later, once there's data.)
- **Consistency is enforced, not hoped for:** because the section is required and shape-checked
  like the dead-ends and last-green sections, the deterministic spec-checker flags a missing or
  malformed corrections log at graduation.

### Scope fence (one sentence)

> Cross-project learnings about the **operator** — your own taste and decision patterns that
> recur across repos, independent of any one domain — are **out of Specline's scope.** They
> belong in your own memory layer (global `CLAUDE.md` / memory), **not** any repo's
> `conventions/`.

## Migration — non-breaking

- C3 clarifies existing rules (checks already commit at ratification); the diff rule ships when
  the diff layer does.
- D is prose.
- The `## Corrections` section is the one new artifact surface; it's small, shaped like the
  existing required sections, and only graduating specs are checked for it.
- The scope fence is prose.

## Cost, honestly

One sharpened boundary (C3), one prose paragraph (D), one required status section that folds
into the permanent record, one scope-fence sentence, one new deterministic doctor check, and
one *(planned)* diff-layer check. All additive, all non-breaking. The deferred items carry zero
cost now and are recorded so they're revisited when a real feature — not a theory — justifies
them.
