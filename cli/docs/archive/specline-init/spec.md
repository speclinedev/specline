# specline init / sync — the scaffolder

## Intent

Build the **writer** half of the `specline` CLI: `specline init` scaffolds a
repo's Specline structure, and `specline sync` regenerates the generated artifacts
from the pinned canon. This is what makes the structure *self-documenting* — every
recommended folder gets a generated `README.md` explaining what it's for and how to
write/read in it — and what keeps those READMEs from drifting when the canon
amends. doctor (`0001-doctor`) validates and reports drift; `init`/`sync` are the
only commands that write generated artifacts. The two never overlap: the validator
stays read-only, the scaffolder writes.

**Appetite — one sitting.** Folder scaffold, the per-folder generated `README.md`
set, `.id-counter` and the `conventions/doc-architecture.md` pin, and idempotent
re-runs. `relations-index.yml` generation rides the same mechanism but is its own
deferred spec.

## Non-goals

- **Never edits authored content.** `init`/`sync` write only *generated* files
  (folder READMEs, and later `relations-index.yml`) and the initial scaffold. They
  never touch a `spec.md`, a knowledge doc, an ADR, or any prose a human or agent
  authored. A generated file is recognizable by its `generated · do not edit`
  header; anything without that header is off-limits.
- **No judgment, no model, no network** — same purity contract as doctor. The
  README content is a deterministic projection of the canon's per-folder
  definitions, not generated prose.
- **Not the validator.** `init`/`sync` do not check conformance; they produce and
  refresh artifacts. Drift detection is doctor's (`README-STALE`), not theirs.
- **Does not migrate content.** It scaffolds structure; moving existing docs into
  `knowledge/` is a human/agent task (see the adoption path), not something `init`
  authors.

## Behavior

1. **`specline init [PATH]`** scaffolds a Specline repo under `PATH/docs/`: the
   recommended directories, each with its generated `README.md`; an empty
   `specs/.id-counter` (`0000`); and a `conventions/doc-architecture.md` carrying
   the canon-version + tier pin. It prints what it created.

2. **Idempotent and non-destructive.** Re-running `init` adds only what's missing
   and refreshes generated files; it never overwrites authored content or an
   existing non-generated file. Safe to run on a populated repo.

3. **`specline sync [PATH]`** regenerates every generated artifact from the
   currently pinned canon — primarily the folder READMEs — and updates their
   header to the new canon version. Run after a canon-version bump. Touches no
   authored content.

4. **Generated-file contract.** Every file `init`/`sync` writes carries a first-line
   header: `<!-- generated · canon <version> · do not edit · run: specline sync -->`.
   The presence of that header is what marks a file regenerable and safe to
   overwrite; its absence marks a file off-limits.

5. **README content is canon-derived.** Each folder's `README.md` is the canon's
   definition of that folder, rendered into the fixed skeleton (What's here / How
   to write / How to read / What's not here). Same canon + same folder → identical
   bytes (determinism).

6. **`--check` (dry run).** `init --check` / `sync --check` print what *would* be
   written and exit non-zero if anything is missing or stale, without writing —
   for CI and pre-commit use. (The authoritative drift gate remains `doctor`.)

7. **Tier-aware.** `init` scaffolds only the folders the declared (or `--tier`)
   tier calls for; a Tier-1 repo does not get Tier-2 directories it won't use.

8. **Optional GitHub Action.** `init` offers to generate the gate workflow
   (`.github/workflows/specline.yml`, which invokes `0003-github-action`): an
   interactive prompt — `Add a GitHub Action to gate docs/ on PRs? [y/N]`,
   defaulting to no — or `--github-action` / `--no-github-action` to answer it
   non-interactively (`--yes` accepts all defaults for unattended runs, so `init`
   never blocks a script on a prompt). The workflow is a generated artifact like
   the READMEs: same header, regenerated by `sync`.

## Business rules

- `init`/`sync` **must not** write any file lacking the generated-artifact header
  except the initial scaffold of a missing recommended file.
- A generated file's content **must** be a pure function of (canon version, folder)
  — no timestamps or machine-specific data in the body — so `sync` is deterministic
  and `doctor`'s drift check is decidable.
- `init`/`sync` **must not** open a network connection or invoke a model.
- `init`/`sync` **must** leave a repo that `doctor` then validates with zero errors
  (a freshly `init`-ed repo is conformant by construction).

## Critical files

- `0001-doctor` — shares the `specline` CLI umbrella, the canon-reading, and the
  output/determinism posture. The drift check (`README-STALE`) is added there.
- Specline canon `2.3.0-draft`, repo `speclinedev/specline` (`specline-2.3.md`)
  § *Repository layout → Self-documenting folders* — the convention this implements.

## Acceptance checks

- *(agent-loopable)* `specline init` on an empty directory produces a `docs/` tree
  that `doctor` validates with zero errors.
- *(agent-loopable)* Every file written by `init`/`sync` begins with the
  generated-artifact header.
- *(agent-loopable)* Two `sync` runs against the same canon produce byte-identical
  files (determinism).
- *(agent-loopable)* `sync` after a canon-version bump updates README bodies and
  headers; `init`/`sync` leave every header-less (authored) file untouched —
  verified by hashing authored files before and after.
- *(agent-loopable)* `init --check` on a repo missing a generated README exits
  non-zero and writes nothing.
- *(agent-loopable)* `init --github-action` writes `.github/workflows/specline.yml`
  (carrying the generated header) and `init --no-github-action` writes none;
  neither prompts under a non-interactive run.
- *(human-gate)* `jonathan` confirms a folder `README.md` reads clearly enough that
  an agent landing in that directory could write a conforming document from it
  alone.

## Out of scope / deferred

- **`relations-index.yml` generation** — the reverse-edge index. Same
  generated-artifact mechanism; its own spec.
- **The `README-STALE` drift rule** in doctor — belongs to doctor's rule set (added
  alongside the lifecycle rules), not built here. This spec defines the header
  contract the check relies on.
- **Content migration** — importing existing docs into `knowledge/` (the adoption
  path) is a separate concern.
