## The unifying insight

The last several threads all live at the **same seam — between *shaped* and
*built/verified*** — and they're one design, not five patches. The canon
under-specified four things at that seam:

1. the **altitude** at which a check is verified (it named *who* verifies, not *how
   settleable* the claim is),
2. the **placement of the reviewer** in the loop (it flattened two loops into one),
3. the **size** of a unit of work (it conflated spec-reviewability with build-size),
4. and the **knobs** that tune all of the above (buried in a markdown table).

Fix the seam once and all four resolve. The threads also interlock: the reviewer's
*bounce budget* and the *slice threshold* are both `specline.yml` knobs; `appetite`
and `blast_radius` both gate loop behavior; the new `judgeable` partition is what the
reviewer attests *after* the inner loop goes green.

## The five changes, as one design

### A. Verification altitude — acceptance has three altitudes, not two

The acceptance partition was named by *verifier identity* (machine vs. human). That
axis has a hole in the middle, and the fresh-context reviewer lives in it. Re-document
the axis as **altitude**:

- **provable** — settled by a runnable command. *(= the existing `agent-loopable`.)*
- **judgeable** — settled by a fresh-context agent **against a named spec section**,
  by no command. *(NEW.)*
- **tasteable** — settled only by the decider, once. *(= the existing `human-gate`.)*

**Keep the field labels `agent-loopable` / `human-gate`** (version-skew discipline —
don't rename known fields); **add `judgeable`** as a third. The axis becomes altitude
in prose; the labels stay.

**Falsifiability, sharpened (B5).** Falsifiable = *settleable the same way twice*, in
one of three forms: by command, by fresh-context judgment **against a named section**,
or by the decider in one sitting. `judgeable` is a *second falsifiable form*, not an
escape hatch — and the section reference is its falsifiability gate, exactly as the
backticked command is `agent-loopable`'s. *"Improve the dashboard"* cites no section,
fails all three, stays draft.

**Honest caveat (state it plainly in the canon):** `judgeable` is a *softer*
falsifiability than `provable` — an agent's judgment is not deterministic. The amendment
introduces **bounded judgment into the loop**. The new posture is therefore:
**deterministic *inner* loop; bounded-judgment *outer* loop; unbounded judgment at the
human end.** Judgment is *layered*, not banished from the middle. We soften the purity
on purpose, in exchange for the reviewer finally having a home.

### B. The two loops — inner (mechanical) + outer (verifier bounce)

The single "build loop" hid two:

- **Inner loop:** implementer ↔ **provable** checks. Mechanical exit (the checks pass),
  bounded by `loop_budget`/`ttl_expires`. The reviewer is correctly absent.
- **Outer loop:** verifier → implementer against the **judgeable** partition. Its own
  small **bounce budget**. It gates the **transition to the human**, never the inner
  loop. This preserves "the reviewer does not hold the build loop" while restoring a
  real second gate — and gives an external orchestrator a legitimate place to put the
  reviewer.

The orchestrator is external/pluggable, so the canon **names** both loops and their
budgets in the runner contract; it does not implement them.

### C. Work size — re-scope One-Sitting; add `appetite`; nudge on mismatch

**Re-scope B1 / One-Sitting:** it constrains the **spec's reviewability** (a human can
read and judge the *contract* in one sitting), **not the build's size**. A large atomic
migration with a tight spec is *eligible*. Build size is governed by `appetite` /
`blast_radius` / `loop_budget`, not by One-Sitting. This *keeps* the PRD-bloat
protection and *removes* the wrong exclusion.

**`appetite` (new frontmatter field)** = the declared *build* size, set at ratification
the way `blast_radius` is declared risk. Values: `small` (default — one slice) |
`large` (an atomic batch).

**Two valid big-shapes**, chosen by one shaping question — *can this ship in
independently-valuable increments?*
- **yes → decompose** (parent map + sequential scopes — see D).
- **no → one large scope** (`appetite: large`, tight spec, big budget, high
  `blast_radius` + verifier). Correctly large, not ineligible.

**doctor nudges on the *mismatch*, never on size itself.** A crude deterministic proxy
(count of acceptance checks / Behavior items) over the configured threshold *while*
`appetite: small` → a warning: *slice it, or declare `appetite: large` if it's atomic.*
doctor raises the question; the human answers; doctor respects the answer. Promotion of
slicing happens **up front in shaping** (a good agent proposes the cuts); the doctor
rule is the deterministic backstop that stops an un-sliced beast passing unexamined.

### D. Decomposition — the parent-map shape

When a feature is too big for one scope, it decomposes into buildable **scopes** plus a
**parent** that is a **map, not a plan**. (Not a "bet" — no Shape Up wager/appetite
baggage, no exploratory connotation. This is *committed* planning, so it lives in
`specs/`, not a separate exploratory home.)

- **The parent holds, once:** intent, shared non-goals, the load-bearing invariant(s),
  external dependencies, and an index of its child scopes (with status rollup — the
  parent ships when its scopes ship).
- **The parent forbids:** Goal-as-single-check, Behavior, acceptance — any mechanics.
  Its balloon-guard, symmetric to One-Sitting: **stay a map.** doctor flags a parent
  that grows mechanics ("decompose into scopes") or a parent with no children ("misfiled
  scope").
- **The loop targets the scopes, never the parent.** A shared external dependency is
  declared once on the parent and **materialized as the first child scope**; downstream
  scopes `depend_on` it. (This resolves the stranded-dependency finding directly.)

Scopes remain the default and the gravity well; the parent appears only when a feature
genuinely splits. A solo planner doing one scope never writes one.

### E. Config surface — `specline.yml` at repo root

Pull the **tunables** out of the `doc-architecture.md` markdown table into a
machine-readable `specline.yml` at repo root. doctor reads it deterministically
(today it regexes the tier out of a table — fragile).

```yaml
# specline.yml
canon: 2.4.0-draft
tier: 1
deciders: [jonathan]
deputy: null
budgets:
  ttl_building_days: 7
  ttl_blocked_days: 14
  decider_building: 3
  decider_active: 6
  context_budget_pct: 50
thresholds:
  slice_acceptance_checks: 6   # SCOPE-EXCEEDS-APPETITE warns above this when appetite: small
  outer_bounce_budget: 2       # verifier↔implementer rounds before escalating to the human
model_tiers:
  light: claude-haiku
  standard: claude-sonnet
  frontier: claude-opus
```

**The boundary:** `specline.yml` tunes **thresholds and pins** — numbers and names a
repo calibrates to its scale. It does **not** configure rules, schema, or partitions.
Tune the *grain*, not the *methodology*.

## New doctor rules (all **warnings** first, while the format settles)

| rule_id | fires when |
|---|---|
| `JUDGEABLE-NO-SECTION` | a `judgeable` acceptance item cites no spec section to verify against |
| `SCOPE-EXCEEDS-APPETITE` | measured size (acceptance/Behavior count) over threshold while `appetite: small` |
| `PARENT-HAS-MECHANICS` | a parent-map carries Behavior / acceptance (it's regressing into a plan) |
| `PARENT-NO-SCOPES` | a parent-map lists no child scopes (it's a misfiled scope) |

Plus: doctor reads pins/thresholds from `specline.yml` (with `doc-architecture.md` /
canon defaults as fallback).

## Migration — non-breaking

- Empty `judgeable` partition is legal (like an empty `human-gate`) → existing two-way
  specs stay valid.
- `appetite` is optional, defaults to `small`.
- The parent-map appears only when decomposing; single scopes are unaffected.
- `specline.yml` is optional; absent → fall back to `doc-architecture.md` + canon defaults.

## Cost, honestly

One new frontmatter field (`appetite`), one new acceptance partition (`judgeable`), one
config file (`specline.yml`), four new doctor warnings, the parent-map shape, and the
inner/outer naming in the runner contract. All additive, all non-breaking. The only
genuinely new *surface* is the outer loop's bounce budget — which is declared in the
external runner contract, so it's a contract addition, not an implementation the canon
owns.
